## HIGH severity

### BUI-137 — record-win partial chunk failure returns HTTP 200; skill reports success and silently drops the failed chunk's wins
- **Area:** `collection-add` · **Category:** failure-handling · **Confidence:** high
- **What's wrong:** cmd_collection_record_win commits in chunks of 25; if a chunk's cache.write_wins raises, it logs the error, sets partial_failure=True, and continues — rows_written only counts the chunks that DID commit (locg-cli/src/locg/commands.py:2069-2081, 2083-2094). The endpoint returns that dict verbatim with HTTP 200 (routes.py:1101 — only RuntimeError is mapped to 500). The skill's Step 3 relies solely on `curl -sf` (non-200 = fail) to detect failure, never inspects `partial_failure`, and its documented success-response example (collection-add.md:92-100) does not even list the field. So a partial commit (e.g. chunk 2 of 3 fails) is read as full success and those wins are never recorded into the collection — they won't be in the export CSV and won't reach LOCG, with no error surfaced to the user.
- **Evidence:** commands.py:2079-2081 `except Exception as exc:\n  logger.error("Chunk commit failed: %s", exc)\n  partial_failure = True`; commands.py:2094 returns `"partial_failure": partial_failure`; routes.py:1100-1103 `try:\n  return cmd_collection_record_win(req.wins)\nexcept RuntimeError as exc:` (no partial_failure handling, no non-200); collection-add.md:80 `POST it ... (curl -sf so a non-200 fails loudly)` and the success block lines 92-100 omit partial_failure entirely.
- **How it bites:** Record a batch >25 wins where a later chunk hits a write error (disk/flock/lock contention). Endpoint returns 200 with rows_written reflecting only earlier chunks and partial_failure:true. Skill prints 'success' and the dropped wins never appear in the CSV — a silent loss of recorded purchases.
- **Suggested fix:** Either (a) in routes.py api_record_win, raise HTTPException(status_code=500/207) when result.get('partial_failure') is True so `curl -sf` halts the skill, or (b) add an explicit Step 3 instruction to abort if the response has partial_failure:true. Prefer (a) so any HTTP caller is protected, not just the skill.
- **Regression test:** Endpoint test: mock cmd_collection_record_win to return partial_failure:True and assert /api/comics/collection/record-win responds non-200 (or the overlay re-raises). Plus a skill-contract test asserting the skill treats partial_failure:true as failure.
- **Verification:** Verified all three legs in current code.

1) commands.py:1910-2081 — `for chunk in chunks:` iterates every chunk; on a write_wins exception the handler at 2079-2081 (`except Exception as exc: logger.error("Chunk commit failed: %s", exc); partial_failure = True`) only logs/sets the flag and the loop CONTINUES. `rows_written` (incremented at 2071 only inside the try) reflects only committed chunks, so a failed middle chunk's wins are lost from the count and from the store. The return dict (2083-2095) carries `partial_failure` but the function does not raise.

2) routes.py:1099-1103 — `api_record_win` does `return cmd_collection_record_win(req.wins)` verbatim and only catches `RuntimeError -> HTTPException(500)`. A normal return with `partial_failure: True` is sent with default HTTP 200. There is no inspection of `partial_failure` and no non-200 path for it.

3) collection-add.md:80,84 — the skill detects failure solely via `curl -sf` (non-200 fails). The documented success block (lines 92-100) does NOT list `partial_failure`, and the "If the POST fails, STOP" note (line 102) only fires on non-200. So a 200 with partial_failure:true is read as full success.

Net: a partial commit (chunk >25 wins, later chunk write error) returns 200, skill reports success, dropped wins never appear in the export CSV (Step 4) and never reach LOCG, with no error surfaced. Not guarded anywhere. Severity is high (silent loss of recorded purchases) though narrow in trigger: requires a batch >25 and a mid-stream write_wins failure; single-chunk record-wins (the common case) are unaffected.

### BUI-138 — Step 2 export: curl failure leaves a stale /tmp/sync-export.json that the very next line happily re-parses into a CSV
- **Area:** `collection-sync` · **Category:** failure-handling · **Confidence:** high
- **What's wrong:** Step 2 fetches the export with `curl -sf "$GIXEN_SERVER_URL/api/comics/collection/export" -o /tmp/sync-export.json` and then, as a SEPARATE unchained command, runs a `python3 -c` that does `json.load(open('/tmp/sync-export.json'))` and writes `d['csv']`/`d['notes_md']` to ~/Downloads. There is no `|| exit` guard on the export curl and no STOP-on-failure prose for Step 2 (contrast Step 5, which explicitly says "If the POST fails, STOP"). I verified curl's `-f`/`-o` behavior: on a connection failure (server down, exit 7) OR an HTTP error like 404/500 (exit 22), curl exits non-zero but LEAVES THE EXISTING OUTPUT FILE UNTOUCHED — it does not truncate or remove /tmp/sync-export.json. So if a prior sync run left a /tmp/sync-export.json behind, and this run's export call fails (the /export route 500s when CollectionCache.load() raises RuntimeError on a corrupt/crashed store — routes.py:1021-1022), the python line silently loads STALE prior-run data and generates a bulk-import CSV from it. That CSV then goes through the manual LOCG upload (Step 3) carrying wrong/old rows — exactly the duplicate-upload / wrong-data hazard the skill's own preamble warns about. Because /tmp is a stable path reused every run, the stale file is the common case, not an edge case.
- **Evidence:** `.claude/commands/comic/collection-sync.md:69-74`: `curl -sf "$GIXEN_SERVER_URL/api/comics/collection/export" -o /tmp/sync-export.json` immediately followed by an unchained `python3 -c "import json,os; d=json.load(open('/tmp/sync-export.json')); ..."` with no `|| exit` between them and no STOP text. Verified runtime: `curl -sf <down-host> -o f.json; echo $?` → 7, file unchanged; `curl -sf <reachable>/404 -o f.json` → 22, file unchanged. The /export route can return 500: `routes.py:1021-1022` `except RuntimeError as exc: raise HTTPException(status_code=500, detail=...)`.
- **How it bites:** Run /comic:collection-sync once successfully (leaves /tmp/sync-export.json). Later run it again when the server's collection store is momentarily unavailable or the host is unreachable: the export curl fails non-zero, but the next line loads the OLD /tmp/sync-export.json and writes a ~/Downloads/locg-bulk-import-<ts>.csv from last run's pending rows. The user uploads stale rows to LOCG.
- **Suggested fix:** Chain the failure: either `curl -sf ... -o /tmp/sync-export.json || { echo 'export failed'; exit 1; }` AND use a fresh unique temp path (e.g. `mktemp`) instead of the fixed /tmp/sync-export.json so a stale file can never be re-read; or combine fetch+parse so a non-200 aborts (e.g. pipe curl stdout straight into python and let `set -o pipefail` propagate). Add explicit "If the export call fails, STOP" prose to Step 2 to match Step 5.
- **Regression test:** Shell/integration test: pre-create /tmp/sync-export.json with known-old JSON, point GIXEN_SERVER_URL at an unreachable host, run the Step 2 snippet, and assert no ~/Downloads/locg-bulk-import-*.csv is produced (and a non-zero exit) rather than a CSV built from the stale file.
- **Verification:** Verified against current code. collection-sync.md:69 runs `curl -sf "$GIXEN_SERVER_URL/api/comics/collection/export" -o /tmp/sync-export.json` with NO `|| exit` and NO STOP prose, then lines 71-74 run a SEPARATE unchained `python3 -c "...json.load(open('/tmp/sync-export.json'))..."` that writes `d['csv']`/`d['notes_md']` to ~/Downloads. I empirically reproduced curl's behavior: against an unreachable host `curl -sf ... -o f.json; echo $?` returns 7 and LEAVES the pre-existing f.json untouched (not truncated/removed); the subsequent `python3 -c "json.load(open('f.json'))"` then parses the STALE valid JSON cleanly and prints the old ready_count, exiting 0 — so a stale CSV is produced from prior-run data. The /export route genuinely can fail with a 500 (routes.py:1021-1022: `except RuntimeError as exc: raise HTTPException(status_code=500, ...)`), and an HTTP error also makes `curl -f` exit non-zero while leaving the output file untouched. The fixed `/tmp/sync-export.json` path is reused every run, so a leftover from a prior successful sync is the common case. This is NOT guarded: Step 0 (line 32) and Step 5 (lines 143-144 'If the POST fails, STOP') both have explicit failure guards, proving the pattern is known and deliberately applied elsewhere but omitted in Step 2. The Step 3 manual review gate does not catch this specific case — a stale CSV carries old valid In Collection=1 rows that produce no 'Deleted from Collection' warning, so the human reviewer sees nothing anomalous and uploads wrong/old rows to LOCG.

### BUI-139 — FMV cache-reuse (_db_lookup) is variant-blind: a base cover can reuse a Newsstand variant's FMV (or vice-versa)
- **Area:** `fmv-linkage` · **Category:** contract-mismatch · **Confidence:** high
- **What's wrong:** BUI-28 made `variant` part of the comic identity: upsert_comic() creates DISTINCT comics rows per variant for the same (title, issue, year), and comic-fmv forwards `variant`/`locg_variant_id` from the batch into POST /api/comics (fmv_runner.py:319-321). But the cache-reuse read path keys ONLY on (locg_id, grade) and never on variant. Two variant rows of the same issue share the same issue-level `locg_id` (only `locg_variant_id` differs), so when both have a fresh FMV at the same grade, _db_lookup picks whichever sorts newest by fmv_updated_at — which may be the wrong variant. The GET /api/comics / list_comics() path also exposes no variant filter and the runner never reads locg_variant_id back, so the caller cannot disambiguate even though the column is returned. Result: a Newsstand variant silently reuses the base cover's FMV (different price tier), producing a wrong bid cap.
- **Evidence:** fmv_runner.py:170-172 `rows = [r for r in rows if r.get("locg_id") == locg_id and r.get("grade") == grade and r.get("fmv_low") is not None]` (no locg_variant_id / variant filter), then `rows.sort(key=lambda r: r.get("fmv_updated_at") or "", reverse=True); return rows[0]`. db.py:1075-1127 list_comics() has no `variant` parameter and returns rows for ALL variants of a (title, issue). upsert_comic() db.py:710 normalizes+scopes by variant so the WRITE creates per-variant rows. The batch shape carries variant (fmv.md:26 `[{... locg_variant_id?, ...}]`) and fmv_runner.py:319-321 forwards it to the upsert.
- **How it bites:** Price a base cover of issue X at 9.4 (writes comic A, fmv low/high). Later price the Newsstand variant of issue X at 9.4 with the same locg_id (writes comic B). Re-run comic-fmv on the base cover within max_age_days: _db_lookup gets both A and B rows (same locg_id+grade), sorts by fmv_updated_at, and may return comic B's (variant) FMV — bidding the base cover at the variant's price.
- **Suggested fix:** Thread locg_variant_id (and/or variant) through the cache-reuse filter: add a `variant`/`locg_variant_id` query param to GET /api/comics→list_comics(), and have _db_lookup pass + re-verify it the same defensive way it re-checks locg_id/grade. When the book has a locg_variant_id, require r.get('locg_variant_id') == locg_variant_id; treat a base (NULL variant) request as matching only NULL-variant rows.
- **Regression test:** In apps/fmv tests: seed two /api/comics rows with identical locg_id+grade but different locg_variant_id and different fmv_low; assert _db_lookup(locg_id=X, grade=9.4, locg_variant_id=base) returns the base row, not the variant. In overlay tests: GET /api/comics?locg_id=X&locg_variant_id=Y returns only the matching-variant row.
- **Verification:** Verified against current code. The WRITE path makes variant part of comic identity: db.py:708-712 normalizes blank variant to NULL and scopes every identity query by variant ("scope every identity query to this variant so reconciliation never crosses variants"), so upsert_comic creates DISTINCT comics rows per variant for the same (title,issue,year). fmv_runner.py:316-321 forwards both locg_variant_id and variant into POST /api/comics. Two variant rows of one issue can therefore share the same issue-level locg_id while differing only in locg_variant_id (fmv.md:342 "locg_variant_id — include only if a separate variant entry was found on LOCG"; collection-check.md:191 shows a Newsstand listing resolving to the canonical/same issue). The READ path is variant-blind: _db_lookup is called with only locg_id+grade (fmv_runner.py:135-136) and filters rows on exactly `r.get("locg_id") == locg_id and r.get("grade") == grade and r.get("fmv_low") is not None` (fmv_runner.py:170-172) — no locg_variant_id/variant check — then `rows.sort(key=lambda r: r.get("fmv_updated_at") or "", reverse=True); return rows[0]` (175-176), returning whichever variant sorts newest. The server cannot disambiguate either: GET /api/comics accepts locg_id but NOT locg_variant_id (routes.py:104-128) and list_comics() has no variant/locg_variant_id parameter (db.py:1075-1083, 1105-1107), though it does SELECT c.locg_variant_id (db.py:1116). The input book even carries locg_variant_id (fmv_runner.py:400) but it is never threaded into the lookup. So when a base cover and a Newsstand variant of the same issue both have a fresh FMV at the same grade and same locg_id, _db_lookup can return the wrong variant's FMV — a different price tier — yielding a wrong bid cap, directly violating the BUI-28 variant-identity contract. No guard exists in the read path; the existing defensive re-check (170-172) covers locg_id/grade only. The precondition (two same-issue variants both freshly priced at the same grade) narrows the real-world frequency, so high (consequence is a wrong money-path bid cap) is appropriate but not critical.
